# Release Process

## Workflow Sequence

```text
push/merge to main  -->  CI (quality gates)  -->  semantic-release  -->  backmerge PRs
                                                              |
                                                              +--> deploy gate (skipped by design)
```

1. CI workflow validates quality and governance checks on `main`.
2. Release workflow runs only after **successful** CI on `main`.
3. Backmerge workflow runs only after **successful** release on `main` and opens PRs to `staging` and `develop`.
4. Deploy workflow is currently a reserved skip-only gate (runs after release).

## Release Automation Behavior

- Releases are generated by `semantic-release`.
- Version bumps follow Conventional Commits (angular preset):
  - `feat!:` or `BREAKING CHANGE:` footer => **major**
  - `feat:` => **minor**
  - `fix:`, `perf:`, `revert:`, `release:` => **patch**
  - `chore(deps):` / `chore(deps-dev):` => **patch**
  - `docs:`, `chore:`, `refactor:`, `test:`, `ci:`, `build:` => no release
- Root `package.json` version is updated during release (`npmPublish: false`).
- GitHub Release notes are generated automatically.

## Requirements

- Conventional commits
- Semantic PR titles
- Green CI checks before merge to `main`

## Local formatting

Run `pnpm format` before commit to apply formatters across the monorepo (ESLint, Ruff, Markdownlint). Husky still auto-fixes staged Markdown on commit.
